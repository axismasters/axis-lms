# CHANGES_PHASE3D.md

## v3-r9 — IF Analysis Engine 계약 정리 + 테스트 상세 흐름 안정화

**버전**: v3-r9 — v3-r8 GitHub main 반영 및 Build Check 통과 확인 완료 상태.
이번 라운드 목표는 화면 재작업이 아니라 "이미 잘 만들어진 IF 채점 기능을
공식적인 엔진 경계로 정리하는 것"이었다.

### 조사 결과 — 대부분 이미 지시 기준을 충족하고 있었다

IF 채점 위치, IF 이유 3개 고정, 학생 직접 입력 금지, 학부모 읽기 전용, 선생님
상담 준비용 요약(자동 브리핑 엔진에 IF 포함) — 이 5가지를 전부 코드로
확인했는데, **전부 이미 v3-r7-r1 이전부터 구현되어 있었다.** 특히 `studentIfRecord.ts`
(Phase 3B)와 `studentIfAnalysis.ts`(Phase 2E/3A-2)는 이미 계산·저장·성장엔진
연동까지 잘 짜여 있었다. 그래서 이번 라운드는 "새로 만들기"가 아니라 "이미
있는 걸 하나의 계약으로 이름 붙이고, 그 계약을 어기는 부분 하나를 고치는"
작업이 됐다.

### 발견한 위반 1건 — TeacherStudentGrowth.tsx의 인라인 계산

"IF 상승 가능성"을 `Math.min(100 - avgPct, 20)`으로 컴포넌트 안에서 직접
계산하고 있었다 — "화면 컴포넌트에 판단/계산 로직 직접 삽입 금지" 기준
위반이다. `ifAnalysisEngine.estimateIfPotentialFromAveragePct()`로 빼냈다.

### IF Analysis Engine 계약 파일 신설

`src/lib/ifAnalysisEngine.ts`를 만들어 (1) 기존 계산 함수들을 재노출하고,
(2) 지시된 입력/출력 계약을 타입으로 명시하고, (3) "예상 보완 포인트"라는
새 개념(가장 비중이 큰 IF 사유 + 그에 맞는 다음 행동 제안 한 줄)을
추가했다. 이 파일 자체가 새 계산 로직을 만든 게 아니라 기존 로직을
감싸는 "계약 레이어"라는 점이 중요하다 — 나중에 문제은행/엠블럼/라이벌/
학부모브리핑/추천 엔진이 IF 데이터가 필요할 때 이 파일 하나만 보면 되게
했다. 대학추천(Recommendation Engine)은 IF를 입력받지 않는다는 점도
명확히 문서화했다(IF=학원 자체 테스트 회고, 대학추천=실제내신·전국모의
같은 공식 성적 — 서로 다른 데이터 계열).

### 확장 방향 8개 엔진 다이어그램

지시된 8개 엔진(Question Bank / Test Template / Assessment / IF Analysis /
Emblem / Rival / Parent Briefing / Recommendation)의 연결 관계를 ASCII
다이어그램 주석으로 `ifAnalysisEngine.ts` 하단에 남겼다. Emblem/Rival/
Parent Briefing/Recommendation은 이미 어느 정도 구현되어 있다는 점,
Question Bank/Test Template은 아직 없고 지금은 TeacherExams 등이 임시로
그 역할을 대신하고 있다는 점을 정리했다.

### 세션 중 추가 요청 — 로그인 화면 히어로 카드

작업 도중 사용자가 로그인 화면 목업 이미지를 첨부하며 이 형식대로
개발해달라고 요청했다. 이미지에서 카드 배경색을 픽셀 단위로 정밀
샘플링한 결과 `#000926`(브랜드보드에 적힌 UI 기준 Navy `#081F4D`보다
훨씬 짙은 톤)이었다 — 이를 로그인 히어로 카드 전용 색으로만 적용하고,
앱 전역 브랜드 Navy 토큰은 건드리지 않았다(§GPT 전달 의견 참조, 만약
전체 앱의 Navy를 이 톤으로 바꾸길 원하면 다음 라운드에 반영하겠다).
"MATH ACADEMY" 라벨, Gold 강조 태그라인, "(자동 로그인)" 문구, Gold
버튼+흰 글씨 등 이미지의 디테일을 그대로 반영했다. 페이지 배경 자체는
계속 밝은 톤이라 "전체 다크 로그인 화면 금지" 원칙은 유지된다.

### 검수 방법론 개선 — 실제 tsc 문법 검사 가능함을 발견

이전 라운드들은 괄호 개수 세기 같은 대리 검증만 가능했는데, 이번에
확인해보니 이 컨테이너에 TypeScript 컴파일러가 전역 설치되어 있어서
`tsc -p tsconfig.app.json --noEmit`을 실제로 실행할 수 있었다. 프로젝트
자체 의존성(react 등)은 여전히 설치할 수 없어 타입 관련 에러는 대량으로
나오지만, **문법 오류(TS1xxx 코드)는 진짜로 0건임을 확인했다** — 그리고
v3-r8 베이스라인과 v3-r9 결과물의 전체 에러 라인 수(379줄)가 완전히
동일해서, 이번에 새로 만든/수정한 파일이 기존 대비 어떤 새로운 문제도
일으키지 않았음을 훨씬 신뢰도 높게 확인할 수 있었다. 상세는
`QA_PHASE3D.md` v3-r9 섹션 참조.

   Phase 2E~3B 시점에 이미 IF 계산/저장/성장엔진 연동 구조가 꽤
   탄탄하게 만들어져 있었다는 걸 확인했다. 다음 Phase에서 "문제은행
   엔진"이나 "시험 템플릿 엔진"을 만들 때도, 새로 만들기 전에 이미
   비슷한 기능이 다른 파일에 흩어져 있는지 먼저 확인하는 걸 권장한다

## v3-r8 — 학생 테스트/학부모 포털 정책 정리 + AXIS 브랜드 UI 고도화

**버전**: v3-r8 — v3-r7-r1은 GitHub main 반영 및 Build Check 통과 확인 완료 상태.
이번 라운드는 (1) 사용자가 첨부한 AXIS 브랜드보드 이미지를 기준으로 브랜드 마크를
실제 구현하고, (2) 전 화면 정책 위반 표현 재검색, (3) 학생/학부모 화면 표현·UI
정리, (4) 앱 전역 색상 토큰을 브랜드보드 기준으로 정리하는 작업이다.

### 0. 시작 계기 — "마크가 실제로 구현되어 있지 않았다"

이번 라운드는 사용자가 AXIS 브랜드보드 이미지를 첨부하며 "마크랑 시그니처 색상이
반영되고 마크도 잘 구현됐으면 좋겠다"고 요청하면서 시작됐다. 코드를 확인해보니
로그인 화면과 4개 포털 레이아웃 헤더가 전부 `GraduationCap`/`BarChart2` 같은 lucide
범용 아이콘을 임시로 쓰고 있었고, 브랜드보드의 실제 마크(대각선 골드 슬래시가
"A"와 "X"를 관통하는 디자인)는 어디에도 구현되어 있지 않았다. 그래서 이번 라운드의
1순위 작업을 "AxisMark/AxisWordmark SVG 컴포넌트를 만들어 앱 전체에 실제 브랜드
마크를 적용하는 것"으로 잡았다. 상세 내용은 `MODIFIED_FILES_PHASE3D.md` v3-r8
섹션 "1. 브랜드 마크 실제 구현" 참조.

### 1. 정책 위반 표현 전수 재검색 (1단계 지시 대응)

지시된 13개 금지 표현(성적/모의고사/수능실전주간루틴/합격률/합격 가능성/합격 보장/
안정 합격/불합격/수납/청구/미납/환불/영수증)을 `src/**/*.{ts,tsx}` 전체에서
재검색했다. 결과: **학생 화면에서 실제로 렌더링되는 위반 사례는 없었다** — 검색된
모든 매치는 정책 준수를 명시하는 코드 주석("⚠ 금지: ...")뿐이었고, 실 UI 텍스트에는
등장하지 않았다. 이는 v3-r7-r1까지의 반복 검수가 이미 충분히 촘촘했다는 뜻으로
판단하고 이번 라운드에서는 이 항목을 "재검증 완료, 코드 변경 없음"으로 종결했다.

### 2~3. 학생 화면 / 대학추천 흐름 정리

- 학생 직접 성적 입력 UI: 이미 제거된 상태를 코드 주석으로 확인(Phase 2E에서 추가된
  후 v3-r7 계열의 어느 시점에 제거됨 — 정확한 시점은 파일 히스토리 없이는 특정 불가).
  재추가하지 않았다.
- 학생 화면 재무/수납/청구/미납/환불/영수증: 전수 검색 결과 노출 없음.
- "성적" → "테스트" 표현: 내비게이션 메뉴는 이미 "테스트"로 되어 있었으나 화면
  내부(카드 제목, 빈 상태 문구)에는 "성적"이 다수 남아 있어 정리했다. 단, 학원이
  직접 주관하지 않는 실제 성적 데이터(실제내신/전국연합모의고사)를 가리키는 문맥은
  "성적"이 의미상 정확하므로 그대로 두었다 — 상세 판단 기준은
  `MODIFIED_FILES_PHASE3D.md` "4. 학생 화면 '성적' → '테스트' 표현 정리" 참조.
- 고1/2 "목표대학추천" vs 고3 "대학추천" 라벨링: 기존 `universityMenuLabel.ts`의
  `detectStudentGradeLevel`로 이미 학년별 분기 처리되어 있음을 확인, 변경 없음.

### 4. 선생님 시험/성적 입력 화면

실제내신성적/전국모의고사가 출제 목록에 없고 선생님 입력 전용으로 분리되어 있는
구조, "수능실전주간루틴" 내부 개념명이 화면에 노출되지 않는 것(전부 "수능실전
모의고사 결과/입력"으로 표시) 모두 v3-r7-r1까지의 상태가 이미 지시 기준을 충족해
변경하지 않았다.

### 5. 학부모 화면 정리

Rival/Emblem/SP/Tier 직접 노출, 상담 기록 원문 노출 모두 전수 검색 결과 없음(주석
뿐). 수납 상태 표시도 이미 "미납 있음"/"미납 없음" 상태 중심으로 되어 있고, 총
청구액/총 미납액 과시형 UI도 v2에서 이미 제거되어 있었다 — 개별 청구서(월별)의
실제 납부에 필요한 청구액/미납액만 남아 있는데, 이는 결제 실행에 필요한 최소
정보라 판단해 유지했다(§GPT 전달 의견 참조, 이견 있으면 조정 가능).

### 6. AXIS 브랜드 UI 고도화 — 이번 라운드 핵심 작업

색상 하드코딩 문제를 다음 3단계로 접근했다.

**(1) 근본 원인 파악**: `index.css`의 `--primary` CSS 변수 자체가 여전히 v3-r7-r1
이전의 구 Indigo-600(`oklch(0.511 0.262 276.966)`)이었다. 과거 "57개 파일 Navy
전역 치환"은 개별 컴포넌트에 흩어진 하드코딩 값을 고친 것이었고, 디자인 시스템의
뿌리인 이 토큰 자체는 아무도 건드리지 않았던 것으로 보인다. 이번에 `--primary`/
`--ring`/`--chart-1`/`--sidebar-primary`/`--sidebar-ring`을 브랜드보드 기준
Navy/Gold로 교체하고, `--brand-navy`/`--brand-gold`/`--brand-ivory` 변수를 신설했다.

**(2) 전면 재검색**: hex 코드(`#7C3AED`, `#8B5CF6`, `#4338CA` 등 인디고/바이올렛
계열)와 OKLCH hue 260~320 범위를 전부 재검색해 약 60곳 이상의 잔존 보라색을
찾아냈다. 이는 이전 라운드의 "전역 치환"이 실제로는 부분적이었다는 뜻이다
(§GPT 전달 의견 참조).

**(3) 문맥별 교체**: 발견된 각 위치를 개별 확인해 의미에 맞는 브랜드색으로
교체했다 — 대학추천 관련은 Gold(RESULT 단계와 의미 연결), Rival 관련·구 primary
링크색은 Navy로 분리해 시각적 구분을 유지했다. 5단계 카테고리 탭(학원평가~
수능실전)처럼 이미 녹색/황색/빨강으로 분화되어 있던 팔레트는 보라색이 아닌
이상 임의로 바꾸지 않았다 — "임의로 새 팔레트를 만들지 말라"는 지시를 "기존에
문제 없던 것까지 단색화하지 말라"는 의미로 해석했다(§GPT 전달 의견 참조, 다른
해석을 원하면 알려달라).

그라데이션/blob/orb 장식은 재검색 결과 실제로는 로그인 화면의 절제된 모서리
포인트, 카드 hover 등 브랜드 톤에 맞는 은은한 사용뿐이었다 — 추가 제거 작업
없음.

강제해(Tailwind `border-collapse` 클래스보다 선택자 특이성이 높아 자동 우선 적용)
sticky th의 하단 border/box-shadow가 스크롤 중 사라지는 버그도 함께 막았다.

(672~768px) 컨테이너를 사용하도록 조정했다. 상담 기록은 카드 스택 대신 컴팩트 테이블
형태로 구성해 관리자/교사 화면답게 스캔하기 쉽게 만들었다. 기존에 이미 완성되어 있던
관리자/학생/교사 화면 전체를 이번에 다시 손대지는 않았다(범위가 너무 커지고, 이미
Phase 3D v1에서 사용성 개선을 마친 화면들이라 회귀 위험이 큼) — 대신 새로 만드는 화면부터
이 원칙을 적용하고, 기존 화면 전체 재설계는 다음 Phase 후보로 남겨둔다(§GPT 전달 의견 참조).

수행했으며, 시험 scope 분리·TEACHER_PRIVATE 권한 로직·학생 재무 차단 로직은 전혀 건드리지
않았다(아래 §7 검증 참조).

## 적용 우선순위 1 — 엠블럼 관리 (`EmblemManagement.tsx`)

- **테이블 헤더 고정**: `<th>` 각각에 `axis-th-sticky axis-th-sticky-56`(관리자 레이아웃 헤더
  56px 기준) 적용. `border-collapse` 테이블이라 border가 사라지는 문제를 피하기 위해
  Phase 2C와 동일하게 `box-shadow: inset` 방식으로 헤더 하단 구분선을 그렸다(tr의
  `borderBottom` 대신 th 자체에 그림자를 줌 — sticky 상태에서도 구분선이 유지됨).
- **엠블럼 등록/수정 팝업 드래그 이동**: 신규 훅 `src/hooks/useDraggableModal.ts` 작성.
  - 팝업은 기본적으로 화면 중앙에서 시작(열릴 때마다 위치 리셋).
  - 제목 영역(`axis-modal-drag-handle`)을 Pointer Capture로 드래그하면 이동.
  - `clampPosition()`으로 각 변 최소 32px는 항상 화면 안에 남도록 제한(완전히 화면 밖으로
    나가지 않음).
  - 뷰포트 폭 768px 미만(모바일)에서는 드래그를 비활성화하고 중앙 정렬만 유지.
  - 팝업 내부 구조를 헤더/바디/푸터로 분리해 헤더·푸터는 고정, 바디만 스크롤되도록 재구성
    (기존에는 팝업 전체가 스크롤되어 내용이 길면 드래그 핸들 자체가 화면 밖으로 밀려나는
    문제가 있었음).
- **클릭 어포던스**: "수정" 텍스트 링크 → `Button` 컴포넌트(outline)로 전환, 숨김 토글 아이콘
  → `Button variant="ghost" size="icon"`로 전환(hover 배경 추가), 활성/비활성 토글 버튼에
  hover/active 트랜지션 추가. 조회 전용 계정에게는 동일한 자리에 순수 텍스트/아이콘만
  보여줘 클릭 가능한 것처럼 보이지 않게 유지.

## 적용 우선순위 2 — 권한 설정 (`PermissionSettings.tsx`)

- 권한 매트릭스 `<thead>`의 `<th>`(기능 컬럼 + 7개 직급 컬럼)에 `axis-th-sticky
  axis-th-sticky-56` 적용. 배경색/하단 구분선(box-shadow)을 th에 직접 지정해 스크롤 시
  헤더가 사라지거나 컬럼 정렬이 깨지지 않도록 했다.
- 체크박스는 네이티브 폼 컨트롤로 이미 클릭 가능함이 명확해 별도 버튼화는 하지 않았다.

## 적용 우선순위 3 — 시험등록 / 문항 추가·삭제 (`AssessmentFormModal.tsx`)

이 모달은 관리자 "시험 등록"과 교사 "내 시험 만들기"가 공유하므로, 여기 반영된 개선은
양쪽 모두에 자동 적용된다.

- **기본 템플릿 2종 추가** (`buildSuneungTemplate()`, `buildNaeshinTemplate()`):
  - **수능형**: 30문항. 1~3번 2점 / 4~13번 3점 / 14~22번 4점 / 23~30번(선택과목) 4점.
    ⚠️ **주의(원장님 확인 필요)**: 이 배점 구조를 문항 수 그대로 반영하면 합계가
    6+30+36+32 = **104점**이 되어, 원 지시서의 "총점 100점" 문구와 정확히 맞지 않는다.
    실제 수능 수학은 공통 22문항 + 선택 8문항 배점이 이보다 더 세분화되어 있어 100점에
    맞춰져 있는데, 이번 지시서에 명시된 4개 구간 배점을 그대로 적용하면 104점이 나온다.
    현재는 **지시서에 적힌 배점 구간을 그대로** 구현해 두었고(임의로 배점을 줄이지
    않음), 각 문항의 배점은 문항 구성 화면에서 자유롭게 수정 가능하다. 정확히 100점
    구조를 원하시면 다음 라운드에서 구간을 조정하겠다.
  - **내신형 대시**: 24문항, 100점 자동 배분(100 = 4점×20문항 + 5점×4문항 — 나머지 4점을
    앞쪽 4개 문항에 1점씩 더 배분).
  - 템플릿 적용 시 문항 type은 모두 '객관식' 기본값(정답은 비워둠 — 교사가 이후 채움).
    교사가 "+ 문항 추가"를 30번 누르지 않아도 기본 구성이 즉시 생성된다.
  - 이미 입력한 정답이 있는 상태에서 템플릿을 누르면(기존 구성을 덮어쓰므로) 확인 모달을
    띄운다.
- **문항 추가/삭제 스테퍼**: "총 N문항 · 만점 M점" 옆에 좌측 "−"(마지막 문항 삭제) / 우측
  "+"(문항 추가) 버튼 구조 추가.
  - 최소 문항 수(`MIN_QUESTIONS = 1`) 아래로는 삭제 불가(버튼 비활성화 + 안내 토스트).
  - 삭제 대상 문항에 입력된 정답이 있으면(입력값 손실 가능성) 확인 모달을 띄우고, 없으면
    바로 삭제.
  - 문항별 개별 삭제(휴지통 아이콘)도 동일한 확인 로직을 타도록 통합(`requestRemoveQuestion`).
- **버그 수정(부수적 발견)**: 이 확인 모달에 `AlertDialog` 컴포넌트를 재사용하는 과정에서,
  `AlertDialog`가 `Dialog` 안에 중첩될 때 확인 팝업의 배경(오버레이)을 클릭하면 클릭
  이벤트가 상위 `Dialog`까지 버블링되어 시험 등록 모달 전체가 함께 닫히는 기존 버그를
  발견했다. `src/components/ui/alert-dialog.tsx`의 배경 클릭 핸들러에 `stopPropagation()`을
  추가해 수정(다른 화면에서 `AlertDialog`를 쓰는 곳에도 동일하게 적용되는 일반적인 수정).

  "상세" 표시는 클릭 어포던스를 보강하는 용도).
- **AssessmentDetail.tsx**: 탭 안의 표 3개(응시자목록/채점현황/결과분석 "학생별 결과") 모두
  sticky header 적용. "결석 처리/결석 취소" 텍스트 링크 → `Button variant="outline"
  size="sm"`, "정정" 텍스트 링크 → 동일하게 버튼화.

## 적용 우선순위 7 — 선생님 내 시험지 관리 (`TeacherExams.tsx`)

  §1 요구사항은 해당 없음으로 판단).

  전체는 클릭 가능한 영역이 아니므로(카드 안의 "채점하기" 버튼만 클릭 가능) 카드 전체에
  hover 효과를 주지 않았다 — 클릭 안 되는 부분에 클릭 느낌을 주지 않는다는 금지 원칙을
  지켰다.

- 불변 파일 4종(`App.tsx`, `TeacherExamGrading.tsx`, `universityAnalysisAdapter.ts`,
  `classData.ts`) MD5: **작업 전후 완전 동일**(아래 QA 문서 참조)

## §8. GPT(개발 총괄)에게 전달할 의견

- 위 §"적용 우선순위 3"에 적은 대로, **수능형 템플릿의 배점 구조(1~3번 2점/4~13번
  3점/14~22번 4점/23~30번 4점)를 그대로 적용하면 합계가 104점**이 되어 "총점 100점"
  요구사항과 어긋난다. 다음 라운드에서 정확한 100점 배점 구간(예: 공통 22문항 74점 +
  선택 8문항 26점 등 실제 수능 방식)을 확정해 주시면 반영하겠다.
- `AlertDialog`(공용 컴포넌트)가 `Dialog` 안에 중첩될 때 배경 클릭으로 상위 모달까지 같이
  닫히는 버그를 이번에 발견·수정했다. 향후 다른 화면에서도 확인 모달을 중첩해서 쓸 일이
  많아질 것으로 예상되어, 이 수정이 공용으로 도움이 될 것으로 판단해 별도 지시 없이
  반영했다.

## 3. PC 웹 최적화 — 핵심 화면 6개 적용

지시된 6개 화면 전부에 데스크톱(`lg:` 1024px 이상) 2~3컬럼 레이아웃을 적용했다. 컨테이너는
`max-w-lg`류 단일 세로 스택에서 `lg:max-w-5xl`~`lg:max-w-7xl`로 확장하고, `lg:grid
lg:grid-cols-3` 기반으로 메인 콘텐츠(2/3 폭)와 요약 패널(1/3 폭)을 나눴다. 모바일은
`space-y-4`(단일 세로 스택)를 그대로 유지해 회귀 없음.

- **TeacherHome.tsx**(`lg:max-w-6xl`) — 인사·관찰 필요 학생은 전체 폭, 좌측(빠른실행/
  숙제/오늘수업) + 우측(미채점시험/최근테스트결과) 2컬럼.
- **TeacherExamGrading.tsx**(`lg:max-w-5xl`, 불변 해제로 이번에 직접 수정) — 좌측(채점
  대기, 넓게) + 우측(채점 완료 요약, 좁게) 2컬럼.
- **TeacherStudentDetail.tsx**(`lg:max-w-6xl`) — 좌측(기본정보/계정관리/브리핑/담당수업/
  출결/최근테스트) + 우측(수업자료/학부모코멘트/상담기록) 2컬럼.
- **StudentHome.tsx**(`lg:max-w-6xl`) — 인사는 전체 폭, 좌측(빠른이동/숙제/최근성적) +
  우측(Rival/성장진열장/목표대학) 2컬럼.
- **StudentGrades.tsx**(`lg:max-w-6xl`, v3-r7의 `lg:max-w-3xl`에서 추가 확장) — 좌측(탭+
  시험목록, 넓게) + 우측(성적 추이 선그래프 2개, 좁게) 2컬럼으로 재구성(기존 2열 차트
  그리드는 세로 스택으로 변경해 좁은 사이드바 폭에 맞춤).
- **ParentHome.tsx**(`lg:max-w-6xl`, v3-r7의 `lg:max-w-4xl`에서 추가 확장) — 좌측(수강반/
  출결/테스트/숙제/수업자료/성장리포트) + 우측(객관지표/상담전확인카드/자녀에게해줄말/
  수납상태/목표대학) 2컬럼.
